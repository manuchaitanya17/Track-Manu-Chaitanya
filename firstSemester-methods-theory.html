Method Anatomy

A method is a named block of reusable logic with a defined contract

The theory describes a method as a block of code that performs a specific task and improves reuse, readability, and modification discipline. Instead of rewriting the same logic repeatedly, you define it once and call it where needed.

That idea becomes concrete through method components: access modifier, return type, method name, parameter list, optional exception list, and the body that actually runs.

SignatureName + parameter list onlyThe theory explicitly notes that a method signature is not the full declaration; it is the identifier plus parameter list.
ContractReturn type shapes caller expectationsA method returning a value must complete with a return statement of the declared type.
Method Types

Predefined methods and user-defined methods solve different layers of work

The theory first distinguishes predefined methods from library classes and user-defined methods written by the programmer. This matters because using a method well starts with understanding who owns it and how it is invoked.

User-defined methods then split further into static methods and instance methods. Static methods belong to the class, while instance methods require an object reference.

StaticCall with the class nameNo object creation is required before invocation.
InstanceCall with an object referenceThe object becomes the path through which the method is invoked.
Overloading

Method overloading keeps one behavior family under one name

Overloading means reusing the same method identifier with different parameter lists. The theory points out two common variants: different argument counts or different argument types.

It also stresses an important restriction: you cannot overload methods by changing only the return type. Binding for overloaded methods happens at compile time by checking the signature.

Static bindingCompile-time method selectionThe compiler chooses the correct overloaded version based on the call signature.
Design habitKeep overloaded methods behaviorally relatedSame name should imply same overall intent, not unrelated work.
Calling Model

Java passes values, even when the value being copied is a reference

The later sections cover stack frames, call order, and Java’s pass-by-value rule. Primitive arguments are copied directly. Object references are also copied, which is why methods can mutate an object through the copied reference but cannot replace the caller’s reference itself.

The theory closes by unpacking the `main()` signature and why `public`, `static`, `void`, and `String[] args` all matter to JVM startup.

Stack memoryEach call gets its own frameMethod parameters, locals, and intermediate values live inside per-call stack frames.
main()The JVM entry point has a fixed shapeThe runtime looks specifically for `public static void main(String[] args)` or acceptable equivalent forms.
