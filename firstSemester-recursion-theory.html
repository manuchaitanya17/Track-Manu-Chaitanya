Foundations

Recursion means a function solves a problem by calling itself on a smaller sub-problem

The theory introduces recursion as a function-call pattern where a method calls itself directly or indirectly. That only works when the big problem can be reduced into smaller instances of the same problem form.

Two requirements are repeated throughout the notes: a base condition that stops the calls, and a recurrence relation that moves the input toward that stop condition.

Base conditionThe stop gateWithout a valid base condition, recursion does not terminate and eventually exhausts stack memory.
RecurrenceThe shrinking stepEach call should move the problem closer to the base condition rather than sideways or backward.
Stack Flow

Recursive calls stack up and return in reverse order

The second theory block walks through how recursion works internally. Each method call gets its own stack frame, waits while deeper calls finish, and then resumes as those calls return back up the chain.

This is why recursive thinking always has two phases: the going-down phase where calls are created, and the coming-back phase where pending work resumes.

Go phaseCalls are added to the stackEach deeper call pauses the older frame until the deeper one returns.
Come phaseReturns unwind in LIFO orderThe deepest successful call returns first, then the next outer one, and so on.
Types of Recursion

Head and tail recursion differ by when the work is done

Tail recursion places the recursive call at the end of the function, so there is no remaining work afterward. Head recursion calls first and performs work while the stack unwinds.

The notes also connect this distinction to the mental model of “go” and “come”: tail recursion tends to do work on the way down, while head recursion surfaces work on the way back.

Tail recursionWork before or with the recursive callUseful when the function’s final action is the next recursive call.
Head recursionWork after the deeper call returnsUseful when the result depends on what happens after the deeper level finishes.
Tradeoffs

Recursion can simplify ideas, but iteration can be cheaper

The theory explicitly compares recursion with iteration. Recursive solutions can be cleaner and more intuitive for structurally recursive problems, but they cost additional stack space for each call.

That means recursion is not automatically better. The right choice depends on problem structure, readability, and whether the stack cost is acceptable.

Simpler modelingUseful for self-similar problemsFactorials, subsequences, and tree-like traversals often map naturally to recursion.
RiskWrong shrinking logic causes stack overflowIf the base condition is unreachable or the recurrence is wrong, stack frames keep piling up.
Supplement

Linear and binary recursion extend the same core idea at different branching widths

The additional PDF is retained as a source resource for this topic. The core distinction is straightforward: linear recursion makes one recursive call at each level, while binary recursion branches into two recursive calls from one state.

That difference changes the recursion tree dramatically. Linear recursion grows like a single path, while binary recursion grows like a branching tree, which affects both reasoning and time complexity.

Linear recursionOne recursive branch per levelExamples include simple countdowns, factorial, and direct single-subproblem recursion.
Binary recursionTwo recursive branches per levelExamples include branching generation problems and some divide-and-combine recursive structures.
