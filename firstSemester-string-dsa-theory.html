Core Foundations

What a Java String actually is?

In the source theory, String is introduced as an object that represents a sequence of characters. It is implemented by the `String` class from `java.lang`, and that matters because it gives you object behavior, built-in methods, and interoperability with interfaces like `Serializable`, `Comparable`, and `CharSequence`.

The same file also frames `CharSequence` as the interface-level idea behind sequential character data. In practical terms, this is why `String`, `StringBuffer`, and `StringBuilder` can all participate in similar text-oriented workflows even though their mutability and threading behavior differ.

Definition String = object, not primitive text It stores a sequence of characters and exposes a method-heavy API for comparison, slicing, searching, and transformation.
Interface Layer CharSequence unifies text containers `String`, `StringBuffer`, and `StringBuilder` all sit under the same character-sequence abstraction.
Memory Management

Literal Creation, Object Creation, and where Strings live?

The theory distinguishes two creation styles. A string literal such as `"Manu Chaitanya"` is checked against the String Constant Pool first. If the same literal already exists, Java reuses that pooled object reference. When the literal is missing, the JVM creates it and stores it inside the pool for later reuse.

By contrast, `new String("Manu Chaitanya")` always creates a new object in heap memory. That is the central memory difference repeated across the theory file: literal-based creation is reuse-friendly, while `new` forces a distinct heap object even when the text is the same.

String Literal Pool-first creation path Java checks the String Constant Pool before allocating a new literal object.
new String() Always creates a heap object Useful when you explicitly want a separate object, but not when reuse is the goal.
Pool and Interning

Why the String Constant Pool matters?

The source document explains the pool as a memory optimization strategy. If many strings share the same literal value, storing one pooled copy is far cheaper than creating repeated duplicates. That is why automatic interning exists for literals and why `intern()` can be called manually when you want a heap-created string to resolve to its pooled equivalent.

This section also supports the classic `==` versus `equals()` distinction. Two pooled references can match under `==`, but two separately created heap objects usually will not. Content checks should still use `equals()` because it compares characters rather than reference identity.

Automatic Interning Literals are reused by default When a literal already exists in the pool, Java returns the old reference instead of allocating again.
Manual Interning `intern()` returns the pooled reference This bridges a heap object back into the pool-based lookup path.
Immutability and GC

Why String immutability keeps showing up?

The theory emphasizes that Strings are immutable: once created, the internal sequence of characters does not change. Operations like concatenation, replacement, and substring extraction create new string objects instead of mutating the original one. This behavior is what makes safe pooling and predictable reuse practical.

The same document also notes the JVM storage shift after Java 7, where pooled strings moved into the regular heap and became eligible for garbage collection. That reduces the long-term memory risk that existed when pooled strings sat in the older fixed-size `PermGen` space.

  • Changing a variable reference is not the same thing as mutating the original String object.
  • Immutability supports safe sharing of pooled literals across many references.
  • GC behavior after Java 7 reduces the risk of permanent pool growth.
